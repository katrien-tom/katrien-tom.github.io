# Golang进阶

# 1.内存逃逸与垃圾回收

# 1.1 内存逃逸（Escape Analysis）

# 1.1.1 什么是内存逃逸？

内存逃逸分析是 Go 编译器在编译阶段进行的一种分析，目的是决定一个变量是否可以分配到栈上。简单来说，栈内存是局部的，一旦超出作用域就会自动释放，而堆内存需要垃圾回收机制来手动清理。

  • 栈分配的优势：
    • 速度快：栈内存的分配和回收是基于指针的。
    • 内存局部性好：栈上的数据更容易被 CPU 缓存。
  • 堆分配的缺点：
    • 速度慢：分配堆内存需要更多的操作，如管理堆空间和垃圾回收。
    • 需要 GC 回收：堆上的内存不能立即回收，需要依赖 GC。

# 1.1.2 逃逸分析的场景

内存逃逸分析主要考虑两种情况：

  1. 局部变量的指针被返回

    • 当你返回一个局部变量的地址时，编译器会将该变量分配到堆上，而不是栈上，因为它会在函数返回后继续存在。
    func f() *int {
    x := 10
    return &x  // 变量 x 逃逸到堆
}

    规范建议：

    • 如果局部变量不需要返回指针，尽量使用值传递。

    • 如果必须返回指针，可以考虑将该变量改为堆上分配（如使用 new() 或使用堆分配的对象）。

      func f() int {
    x := 10
    return x  // 返回值传递，x 不会逃逸
}

  2. 闭包引用外部变量

    • 如果一个闭包引用了外部的局部变量，编译器会将这个变量逃逸到堆上。
    func f() func() {
    x := 10
    return func() {
        fmt.Println(x)  // x 被闭包引用，逃逸
    }
}

    规范建议：

    • 如果可能，避免闭包引用外部局部变量，尤其是当闭包的生命周期较长时。
    • 使用值传递而非指针传递，减少不必要的内存分配。
    • 通过参数传递来避免引用外部变量

    优化示例：

    func f() func() {
    x := 10
    return func(x int) {
        fmt.Println(x)  // 闭包不再引用外部变量，x 被传递为值
    }
}

  3. 接口类型赋值

  • 当你将一个局部变量赋值给接口时，变量的类型和值会一起被复制到堆上。
func f() interface{} {
    s := "hello"
    return s  // 字符串 s 赋值给接口，逃逸到堆
}

规范建议：

  • 尽量避免将局部变量赋值给接口，尤其是在高频调用的场景中。
  • 如果局部变量不再被使用，可以直接传递值类型，避免额外的堆分配。

优化示例：

func f() string {
    s := "hello"
    return s  // s 在栈上，减少堆分配
}
  1. 切片扩容
  • 当切片在运行时发生扩容，编译器可能会将原本应该在栈上的切片分配到堆上。
func appendSlice() []int {
    s := []int{}
    for i := 0; i < 1000; i++ {
        s = append(s, i)  // s 可能逃逸到堆
    }
    return s
}

规范建议：

  • 在确定切片需要较大容量时，可以预先为切片分配足够的容量，避免频繁扩容。
  • 使用 make 为切片预分配一定的容量，减少切片的扩容操作。

优化示例：

func appendSlice() []int {
    s := make([]int, 0, 1000)  // 预分配容量，避免扩容
    for i := 0; i < 1000; i++ {
        s = append(s, i)  // 不会发生逃逸
    }
    return s
}
  1. 大对象
  • 当对象非常大时，编译器会避免将其分配在栈上，因为栈的大小是有限的。即使没有显式逃逸，编译器也会将它们分配到堆上。
func largeObject() *[1024]int {
    var large [1024]int
    return &large  // 大数组直接逃逸到堆
}

规范建议：

  • 如果大对象确实需要在函数间传递，可以使用指针传递。
  • 对于大对象，如果频繁使用，可以考虑使用 sync.Pool 或 new() 来管理内存。

优化示例：

func largeObject() [1024]int {
    var large [1024]int
    return large  // 直接返回大对象，避免堆分配
}

# 1.1.3 如何查看逃逸分析

使用 -gcflags="-m" 来查看哪些变量发生了逃逸。

go build -gcflags="-m" main.go

输出示例：

./main.go:6:6: x escapes to heap

# 1.2 垃圾回收（Garbage Collection）

# 19.2.1 GC 背景与原理

Go 的垃圾回收机制基于 三色标记清除（Mark-Sweep）算法，并结合 并发 和 分代收集 的优化技术。

# 1.2.1.1 三色标记法

GC 将堆中的对象分为三种颜色：

  • 白色：未被标记，可能是垃圾。
  • 灰色：可达的对象，但其子对象尚未标记。
  • 黑色：可达的对象，且所有子对象已标记。
# 1.2.1.2 标记-清扫过程
  • 标记（Mark）阶段：从根对象（栈、全局变量等）出发，遍历并标记所有可达的对象为灰色，随后标记为黑色。
  • 清扫（Sweep）阶段：扫描所有对象，回收所有仍为白色的对象（不可达的垃圾）。

# 1.2.2 GC 的运行时行为

# 1.2.2.1 并发 GC

Go 的 GC 会与程序并发运行，通过分代收集减少 GC 停顿时间。主要分为以下几个阶段：

  1. STW（Stop-the-world）：开始时需要停止所有 goroutine 进行标记。
  2. 标记阶段：并发执行，程序继续运行。
  3. 清扫阶段：标记完之后，清扫无用的对象。
# 1.2.2.2 GC 停顿
  • 在标记和清扫的某些阶段，可能会发生停顿（Stop-the-World），但是 Go 在设计上尽量减少这些停顿的时间。
  • GC 停顿的时间会影响程序的性能，尤其是大规模的并发系统中。
# 1.2.2.3 Go 1.5+ 的并发收集优化
  • Go 从 1.5 版本开始引入了 并发标记 和 分代式 的垃圾回收优化。每次 GC 都会尽量在并发模式下执行，减少停顿时间。

# 1.2.3. GC 优化技巧

# 1.2.3.1 减少逃逸分析
  • 减少堆分配：尽量使变量在栈上分配。
    • 使用值传递而不是指针传递。
    • 避免返回局部变量的指针。
# 1.2.3.2 使用 sync.Pool 来复用对象
  • sync.Pool 是一个内存池，用于复用临时对象，减少 GC 压力。
var bufferPool = sync.Pool{
    New: func() interface{} { return make([]byte, 1024) },
}

func handler() {
    buf := bufferPool.Get().([]byte)
    defer bufferPool.Put(buf)
    // 使用 buf
}
# 1.2.3.3 减少临时对象的创建
  • 在高频调用的函数中，避免创建临时对象。比如，不要频繁地创建大数组或大切片。
# 1.2.3.4 使用 GODEBUG=gctrace=1 查看 GC 的细节
  • 在开发中可以通过 GODEBUG=gctrace=1 查看垃圾回收的详细信息，以便优化程序。
GODEBUG=gctrace=1 go run main.go

输出示例：

gc 1 @0.086s 3%: 0+0+0=0

# 1.3 内存逃逸与 GC 在实际开发中的应用

# 1.3.1 逃逸分析与 GC 的关系

  • 内存逃逸分析和 GC 关系密切。逃逸分析决定了一个对象是否会分配在堆上，而 GC 负责回收堆内存。
  • 逃逸分析能有效减少堆内存分配，降低 GC 的压力。

# 1.3.2 高效的内存管理

在开发过程中，你可以结合以下几种策略来提升性能：

  • 尽量减少逃逸，让更多的变量分配到栈上。
  • 使用内存池（sync.Pool）减少 GC 压力。
  • 进行 GC 监控，及时优化内存使用。

# 2. 单元测试（Unit Testing）

# 2.1 定义 / 概念

  • 单元测试是指对最小可测试单元（函数/方法）进行正确性验证的测试。
  • Go 内置了 testing 包，支持直接编写和运行单元测试。
  • 通过 go test 命令执行测试代码，快速检测逻辑是否符合预期。

# 2.2 适用场景 / 应用

  • 验证函数逻辑是否正确
  • 保证代码修改后不破坏已有功能（回归测试）
  • 自动化测试，提升开发效率和代码质量
  • 常见场景：
    • 算法函数的正确性验证
    • 工具库的稳定性测试
    • HTTP 接口逻辑测试（配合 httptest 包）

# 2.3 原理 / 思路

  • 在 Go 中，测试文件命名为 xxx_test.go
  • 测试函数必须以 Test 开头，签名为 func (t *testing.T)
  • 使用 t.Errorf / t.Fatalf 等方法报告错误
  • 支持子测试（t.Run）和表格驱动测试

# 2.4 基本用法 / 实现方式

  1. 创建 xxx_test.go 文件
  2. 编写以 Test 开头的测试函数
  3. 在终端运行 go test 或 go test -v

示例：

// math_test.go
package math

import "testing"

func TestAdd(t *testing.T) {
    result := Add(2, 3)
    if result != 5 {
        t.Errorf("expected 5, got %d", result)
    }
}

运行：

go test
go test -v   # 显示详细日志

# 2.5 注意事项 / 易错点

  • 测试文件必须以 _test.go 结尾
  • 测试函数必须以 Test 开头且参数为 *testing.T
  • 避免测试间相互依赖（应保持独立性）
  • 大量重复输入输出场景适合用表格驱动测试
  • 注意测试覆盖率：go test -cover

# 2.6 进阶 / 变种

  • 基准测试（Benchmark）：函数签名 func BenchmarkXxx(b *testing.B)，用于性能测试
  • 示例测试（Example）：函数签名 func ExampleXxx()，可生成文档示例
  • mock 测试：通过接口和模拟实现替代真实依赖
  • 测试覆盖率：go test -coverprofile=coverage.out + go tool cover -html=coverage.out

# 2.7 示例代码

// 表格驱动测试
func TestIsEven(t *testing.T) {
    tests := []struct {
        input int
        want  bool
    }{
        {2, true},
        {3, false},
        {0, true},
    }

    for _, tt := range tests {
        t.Run(fmt.Sprintf("input=%d", tt.input), func(t *testing.T) {
            got := IsEven(tt.input)
            if got != tt.want {
                t.Errorf("input=%d: expected %v, got %v", tt.input, tt.want, got)
            }
        })
    }
}

# 2.8 总结

  • Go 自带 testing 包，天然支持单元测试，无需额外依赖
  • 单元测试命名规范：文件 _test.go，函数 TestXxx
  • 注意区分单元测试 / 基准测试 / 示例测试
  • 配合 CI/CD，单元测试能保证代码迭代时的稳定性

# 3. 单元测试覆盖率

# 3.1 定义 / 概念

  • 覆盖率（Coverage）：指测试代码在运行过程中执行了多少比例的源代码。
  • Go 提供内置工具收集和分析覆盖率：go test -cover。

# 3.2 适用场景 / 应用

  • 衡量单元测试的完整性
  • 发现未被测试覆盖的代码路径
  • 提升测试质量，减少遗漏
  • 应用于：
    • 核心逻辑函数的健壮性验证
    • CI/CD 流程中的质量门槛

# 3.3 原理 / 思路

  • 编译时插桩：在代码中插入统计逻辑，记录哪些语句被执行
  • 运行时收集执行信息
  • 生成覆盖率报告（可输出文本或 HTML 可视化）

# 3.4 基本用法 / 实现方式

go test -cover          # 显示覆盖率百分比
go test -coverprofile=coverage.out   # 输出覆盖率文件
go tool cover -func=coverage.out     # 查看每个函数的覆盖率
go tool cover -html=coverage.out     # 生成可视化报告（浏览器打开）

# 3.5 注意事项 / 易错点

  • 高覆盖率不代表高质量：逻辑分支未验证也可能被覆盖
  • 应结合断言和边界情况测试
  • 测试覆盖率是「辅助指标」，不是唯一目标

# 3.6 进阶 / 变种

  • 在 CI/CD 流程中强制覆盖率阈值
  • 使用 -race 参数结合竞态检测
  • 结合 Mock 提升复杂系统覆盖率

# 3.7 示例代码

go test ./... -coverprofile=coverage.out
go tool cover -html=coverage.out

# 3.8 总结

  • 覆盖率是度量工具，不能代替测试质量
  • 建议结合「表格驱动测试 + 边界条件」提升覆盖率
  • 覆盖率报告是优化测试用例的好助手

# 4. 子测试

# 4.1 定义 / 概念

  • 子测试（Subtests）：Go testing 包提供的机制，允许在一个测试函数中定义多个子用例。
  • 通过 t.Run(name, func(t *testing.T)) 创建子测试。
  • 子测试运行时，会显示层级化的测试名称。

# 4.2 适用场景 / 应用

  • 表格驱动测试：批量测试不同输入输出组合
  • 分组测试：将相关用例组织在一起，便于管理
  • 选择性执行：通过 go test -run 运行特定子测试
  • 层次化结构：复杂逻辑可以嵌套子测试，便于组织

# 4.3 原理 / 思路

  • t.Run 会注册一个子测试，并在测试执行时单独运行
  • 子测试之间默认是顺序执行（除非加上 t.Parallel()）
  • 错误会记录在子测试上下文中，不会影响其他子测试

# 4.4 基本用法 / 实现方式

示例：

func TestMath(t *testing.T) {
    t.Run("Add", func(t *testing.T) {
        if Add(2, 3) != 5 {
            t.Errorf("Add failed")
        }
    })

    t.Run("Multiply", func(t *testing.T) {
        if Multiply(2, 3) != 6 {
            t.Errorf("Multiply failed")
        }
    })
}

运行：

go test -v -run TestMath

输出：

=== RUN   TestMath
=== RUN   TestMath/Add
=== RUN   TestMath/Multiply
--- PASS: TestMath (0.00s)
    --- PASS: TestMath/Add (0.00s)
    --- PASS: TestMath/Multiply (0.00s)

# 4.5 注意事项 / 易错点

  • 子测试共享父测试的上下文（全局变量要小心）
  • 循环中定义子测试时要注意 变量捕获问题（需要传值避免闭包陷阱）
  • 如果子测试较多，可以配合 t.Parallel() 并行运行

# 4.6 进阶 / 变种

  • 表格驱动子测试（结合 for 循环和 t.Run）
  • 并行子测试：在子测试中调用 t.Parallel()
  • 嵌套子测试：在子测试中再调用 t.Run，形成树形层级

# 4.7 示例代码

表格驱动 + 子测试：

func TestIsEven(t *testing.T) {
    tests := []struct {
        name  string
        input int
        want  bool
    }{
        {"even number", 2, true},
        {"odd number", 3, false},
        {"zero", 0, true},
    }

    for _, tt := range tests {
        tt := tt // 避免闭包变量捕获问题
        t.Run(tt.name, func(t *testing.T) {
            got := IsEven(tt.input)
            if got != tt.want {
                t.Errorf("expected %v, got %v", tt.want, got)
            }
        })
    }
}

# 5. 基准测试（Benchmark）

# 5.1 定义 / 概念

  • 基准测试（Benchmark）：用于测量函数或代码片段的性能（执行速度和资源消耗）。
  • Go 内置 testing 包支持基准测试，函数以 Benchmark 开头，参数为 b *testing.B。
  • 执行时，Go 会自动多次运行测试函数以获得稳定结果。

# 5.2 适用场景 / 应用

  • 衡量函数或算法的性能差异
  • 比较不同实现方式的优劣（如切片操作 vs 数组操作）
  • 优化代码时，评估改动效果
  • 在性能敏感的场景（高并发、I/O 密集型应用）中验证实现方案

# 5.3 原理 / 思路

  • 基准测试函数签名必须是：
    func BenchmarkXxx(b *testing.B)

  • Go 框架会自动调整循环次数 b.N 以保证测试时间足够长，结果更稳定

  • 通常代码逻辑写在 for i := 0; i < b.N; i++ { ... } 中

  • 结果以 ns/op（每次操作耗时） 显示

# 5.4 基本用法 / 实现方式

示例：

func BenchmarkAdd(b *testing.B) {
    for i := 0; i < b.N; i++ {
        _ = Add(2, 3)
    }
}

运行基准测试：

go test -bench=.
go test -bench=Add

输出示例：

BenchmarkAdd-8   	1000000000	         0.300 ns/op

解释：

  • BenchmarkAdd-8 表示在 8 个逻辑 CPU 上运行
  • 1000000000 表示运行次数（b.N）
  • 0.300 ns/op 表示平均每次调用耗时

# 5.5 注意事项 / 易错点

  • 必须用 b.N 控制循环次数，否则结果无效
  • 基准测试不要包含 I/O、网络等不稳定因素（除非就是测试它们）
  • 注意 CPU 缓存、编译器优化可能导致结果偏差
  • 测试不同实现时，应确保逻辑等价

# 5.6 进阶 / 变种

  • 并行基准测试：b.RunParallel 支持并发性能测试
  • 内存分配测试：b.ReportAllocs() 显示每次操作的内存分配次数
  • 子基准测试：结合 b.Run 测试不同输入参数下的性能
  • 与 pprof 结合：go test -bench . -benchmem -cpuprofile=cpu.out

# 5.7 示例代码

  • 基准测试是性能优化的必备工具
  • 关键点：函数签名 BenchmarkXxx(b *testing.B)，循环次数用 b.N
  • 可扩展到并行、内存分配分析
  • 与 pprof 等工具结合，可进行深入性能剖析

# 6. init() 函数

# 6.1 定义 / 概念

  • init() 是 Go 语言中一种特殊函数，用于包级别初始化。
  • 在程序运行时，包被导入后会自动执行 init()，无需手动调用。
  • 每个包可以有多个 init() 函数（同一包内不同文件的 init() 也会被执行）。

# 6.2 适用场景 / 应用

  • 初始化全局变量
  • 配置文件或环境变量读取
  • 注册驱动、插件、日志等全局资源

# 6.3 原理 / 思路

  • Go 编译器在包加载阶段自动调用 init()
  • 执行顺序：
    1. 同一文件中，按出现顺序执行多个 init()
    2. 不同文件，按编译顺序执行（同一包内）
    3. 依赖包的 init() 在导入包前执行
  • main.init() 在所有依赖包初始化完成后执行

# 6.4 是否是最先执行的函数

  • init() 并非程序中最先执行的函数
  • 执行顺序：
    1. 先执行依赖包的全局变量初始化（变量声明和赋值）
    2. 依赖包的 init() 函数
    3. 当前包的全局变量初始化
    4. 当前包的 init() 函数
    5. main() 函数
  • 因此，init() 是包级初始化的第一步，但不是整个程序启动的绝对第一步

# 6.5 基本用法 / 示例

package main

import "fmt"

var globalVar = initVar()

func initVar() int {
    fmt.Println("Global variable initialization")
    return 10
}

func init() {
    fmt.Println("Init function called")
}

func main() {
    fmt.Println("Main function called")
    fmt.Println("globalVar =", globalVar)
}

输出：

Global variable initialization
Init function called
Main function called
globalVar = 10

# 6.6 注意事项 / 易错点

  • init() 不接受参数，也没有返回值
  • 不要在 init() 中写复杂逻辑，避免难以调试
  • init() 的执行顺序受包导入顺序和全局变量初始化顺序影响
  • 不要假设 init() 总是最先执行

# 6.7 进阶 / 变种

  • 多个 init() 可以组合使用，按文件顺序执行
  • 配合全局变量，实现懒初始化
  • 常用于插件注册模式：
func init() {
    RegisterPlugin("example", &ExamplePlugin{})
}

# 6.8 总结

  • init() 是包级初始化函数，自动调用，无需手动调用
  • 全局变量初始化先于 init() 执行
  • main() 函数在所有 init() 执行完后才运行
  • 使用时注意依赖顺序和执行时机，避免复杂逻辑

func WalkDir(path string) error {
	//filepath.Walk 会递归地遍历一个目录
	filepath.Walk(path, func(subPath string, info os.FileInfo, err error) error {
		if err != nil {
			return err
		} else if info.Mode().IsDir() && subPath != path {
			fmt.Printf("path is dir %s\n", subPath)
		} else if info.Mode().IsRegular() {
			fmt.Printf("path is file %s basename %s\n", subPath, info.Name()) //basename表示去除目录，仅保留文件名，subPath里包含了basename
		}
		return nil
	})
	return nil
}

package io

import (
	"io"
	"log"
	"os"
	"path"
	"path/filepath"
	"strconv"
)

func SplitFile(infile string, outDir string, n int) {
	fin, err := os.Open(infile)
	if err != nil {
		log.Panic(err)
	}
	defer fin.Close()

	stat, err := fin.Stat()
	if err != nil {
		log.Panic(err)
	}
	fileSize := stat.Size()
	chunk := fileSize / int64(n)
	if chunk <= 0 {
		panic("file is too small or n is too large")
	}
	for i := 0; i < n; i++ {
		fout, err := os.OpenFile(path.Join(outDir, strconv.Itoa(i)+"_"+path.Base(infile)), os.O_CREATE|os.O_TRUNC|os.O_WRONLY, os.ModePerm)
		if err != nil {
			log.Panic(err)
		}
		need := int(chunk)
		if i == n-1 {
			need = int(fileSize) - (n-1)*int(chunk)
		}
		buffer := make([]byte, need)
		_, err = fin.Read(buffer)
		if err != nil {
			log.Panic(err)
		}

		_, err = fout.Write(buffer)
		if err != nil {
			log.Panic(err)
		}
		fout.Close()
	}
}

func AppendFile(fout *os.File, infile string) {
	fin, err := os.Open(infile)
	if err != nil {
		log.Panic(err)
	}
	defer fin.Close()

	buffer := make([]byte, 1024)
	for {
		n, err := fin.Read(buffer)
		if err != nil {
			if err == io.EOF {
				if n > 0 {
					fout.Write(buffer[:n])
				}
			} else {
				log.Println(err)
			}
			break
		} else {
			fout.Write(buffer[:n])
		}
	}
}

// 把dir这个目录下的所有文件合并到mergedFile里去
func MergeFile(dir string, mergedFile string) {
	fout, err := os.OpenFile(mergedFile, os.O_CREATE|os.O_APPEND|os.O_WRONLY, os.ModePerm)
	if err != nil {
		log.Panic(err)
	}
	defer fout.Close()

	// ReadDir()得到的子文件/子目录是按文件名排好序的
	if fileInfos, err := os.ReadDir(dir); err != nil {
		log.Panic(err)
	} else {
		for _, fileInfo := range fileInfos {
			if fileInfo.Type().IsRegular() {
				infile := filepath.Join(dir, fileInfo.Name())
				AppendFile(fout, infile)
			}
		}
	}
}

# 8. 高级IO功能

# 8.1 LimitReader

# 8.1.1 作用

​ 返回一个新的 Reader，从原始 Reader 中最多读取 n 个字节。超过部分会被丢弃。

# 8.1.2 用途场景

  • 限制读取数据的大小（防止恶意客户端发送超大请求）

  • 控制文件或网络流的读取上限

# 8.1.3 原理

LimitReader 返回一个包装结构 limitedReader，内部维护一个剩余字节数 n，每次 Read() 时只从底层读取 min(n, len(p)) 个字节。

示例

func limitReader() {
	reader := strings.NewReader("daqiaoqiao")
	limitReader := io.LimitReader(reader, 6) //limitReader截取了reader的前6个字节
	content := make([]byte, 100)
	if n, err := limitReader.Read(content); err == nil {
		fmt.Printf("read %s\n", string(content[:n])) //daqiao
	}
	if _, err := limitReader.Read(content); err == io.EOF { //从limitReader里已读不出任何内容
		fmt.Println("no more data available")
	}
}

# 8.2 MultiReader

# 8.2.1 作用

​ 将多个 Reader 串联起来，形成一个“连续流”。

# 8.2.2 用途场景

  • 把多个来源的内容拼成一个流读取

  • 比如拼接多个文件、多个内存块

# 8.2.3 原理

内部保存一个 []Reader 切片，按顺序调用各个 Read()，当前 reader 读完（返回 EOF）后自动切换到下一个。

示例

func multiReader() {
	r1 := strings.NewReader("黄梅时节家家雨\n")
	r2 := strings.NewReader("青草池塘处处蛙\n")
	r3 := strings.NewReader("有约不来过夜半\n")
	r4 := strings.NewReader("闲敲棋子落灯花\n")

	r := io.MultiReader(r1, r2, r3, r4) //注意，这里是有序的
	io.Copy(os.Stdout, r)               //把r流拷贝到标准输出流
	//借助于MultiReader，可以把多个文件合并成一个文件
}

# 8.3 MultiWriter

# 8.3.1 作用

​ 把写入的数据复制到多个 Writer，类似“广播”。

# 8.3.2 用途场景

  • 同时写到多个目标（例如：文件 + 控制台 + 网络）

  • 实现日志多路输出（文件日志 + 标准输出）

# 8.3.3 原理

​ MultiWriter 内部维护多个 writer，调用 Write(p) 时会依次写入所有 writer。如果有一个写入错误，则立即返回错误。

示例

func multiWriter() {
	var (
		writer1 bytes.Buffer
		writer2 bytes.Buffer
	)
	multiWriter := io.MultiWriter(&writer1, &writer2)
	multiWriter.Write([]byte("黄梅时节家家雨\n"))

	fmt.Print(writer1.String()) //黄梅时节家家雨
	fmt.Print(writer2.String()) //黄梅时节家家雨
	//借助于MultiWriter可以把一条日志输出到多个文件里面去
}

# 8.4 TeeReader

# 8.4.1 作用

TeeReader 会返回一个新的 Reader，它在从原始 Reader 中读取数据时，会 自动把读取到的数据同时写入指定的 Writer。

# 8.4.2 用途场景

  • 调试-》读取 HTTP body 的同时，把内容写入日志

  • 备份-》读取文件流的同时写入另一个目标

  • 哈希计算-》读取数据并计算哈希，而不破坏原始流

# 8.4.3 原理

func teeReader() {
	var writer bytes.Buffer
	reader := strings.NewReader("黄梅时节家家雨\n")
	teeReader := io.TeeReader(reader, &writer) //从reader里读取的内容既会进入teeReader，也会进入writer
	// io.Copy(os.Stdout, reader)  如果打开此行，则reader里的内容已经被读完了，下面2行不会输出任何内容
	io.Copy(os.Stdout, teeReader) //如果把此行注释掉，则没有任何数据经过teeReader，writer里也不会有任何内容
	fmt.Print(writer.String())
}

# 8.5 PipeIO

# 8.5.1 作用

​ 创建一个同步的内存管道（pipe），Reader 从管道读取，Writer 写入管道。可用于连接生产者和消费者 goroutine。

# 8.5.2 用途场景

  • 在内存中连接两个 goroutine 的数据流

  • 实现类似 Unix 管道的机制（生产-消费模型）

# 8.5.3 原理

Pipe 使用内存缓冲区 + 同步锁控制：

  • 写入方阻塞直到读方读取
  • 读方阻塞直到写方写入
  • 适合流式通信（无须中间文件）

示例

func pipeIO() {
	reader, writer := io.Pipe() //writer的内容会直接进入reader，中间没有buffer
	go func() {
		writer.Write([]byte("hello")) //因为中间没有buffer，所以Write操作会阻塞，直到另一个协程准备从reader里读取内容
		writer.Close()
	}()
	content := make([]byte, 100)
	if n, err := reader.Read(content); err == nil {
		fmt.Printf("read %s\n", string(content[:n])) //hello
	}
	reader.Close()
}

	GZIP                 //2

func getCompressWriter(compressAlgo int, file *os.File) (Writer, error) { //返回一个接口
	switch compressAlgo {
	case ZLIB:
		return zlib.NewWriter(file), nil
	case GZIP:
		return gzip.NewWriter(file), nil
	default:
		return nil, fmt.Errorf("INVALID compress algo %d", compressAlgo)
	}
}

func getDecompressReader(compressAlgo int, file *os.File) (io.ReadCloser, error) {
	switch compressAlgo {
	case ZLIB:
		return zlib.NewReader(file)
	case GZIP:
		return gzip.NewReader(file)
	default:
		return nil, fmt.Errorf("INVALID decompress algo %d", compressAlgo)
	}
}

// 解压
func decompress(compressAlgo int, inFile, outFile string) {
	fin, err := os.Open(inFile)
	if err != nil {
		fmt.Println(err)
		return
	}
	stat, _ := fin.Stat()
	fmt.Printf("压缩后文件大小 %dB\n", stat.Size())

	reader, err := getDecompressReader(compressAlgo, fin) //解压
	if err != nil {
		fmt.Println(err.Error())
		return
	}

	fout, err := os.OpenFile(outFile, os.O_CREATE|os.O_TRUNC|os.O_WRONLY, 0666)
	if err != nil {
		fmt.Println(err)
		return
	}

	io.Copy(fout, reader) //把一个流拷贝到另外一个流
	reader.Close()
	fin.Close()
	fout.Close()
}

# 10. log和slog

  • log：传统、简单、非结构化，适合小型程序或脚本。

  • slog：现代、结构化、可扩展，适合生产级系统与日志分析平台。

# 10.1 log 包（传统日志）

# 10.1.1 基本介绍

  • Go 自 1.0 起内置的基础日志包。
  • 仅支持简单文本输出，没有日志等级或结构化能力。

# 10.1.2 基本用法

import (
    "log"
    "os"
)

func main() {
    log.Println("Hello, world")
    log.Printf("User %s logged in", "Tom")

    // 自定义输出
    file, _ := os.Create("app.log")
    logger := log.New(file, "INFO: ", log.Ldate|log.Ltime|log.Lshortfile)
    logger.Println("Log to file")
}
// 输出
// INFO: 2025/10/16 15:04:03 main.go:10: Log to file
特性描述
✅ 简单稳定使用方便，输出到控制台或文件
⚙️ 可定制前缀支持 log.New()
🚫 无日志等级无 Info/Warn/Error 区分
🚫 无结构化字段仅文本日志
🚫 无 JSON 输出不能直接输出 JSON
🚫 无上下文信息不支持属性或标签

# 10.2 slog

# 10.2.1 基本介绍

  • Go 1.21 引入的新标准日志库。
  • 结构化日志（Structured Logging），支持等级、字段、上下文。
  • 类似第三方库 Zap / Zerolog，但更轻量、标准化。

# 10.2.2 基本用法

import "log/slog"

// 1. 输出（默认 text handler）：
func main() {
    slog.Info("server started", "port", 8080)
    slog.Warn("cache miss", "key", "user:123")
    slog.Error("failed to connect", "error", "timeout")
}
// 输出
time=2025-10-16T15:10:20.123Z level=INFO msg="server started" port=8080
time=2025-10-16T15:10:20.124Z level=WARN msg="cache miss" key=user:123
time=2025-10-16T15:10:20.125Z level=ERROR msg="failed to connect" error=timeout

// 2. JSON
func main() {
    handler := slog.NewJSONHandler(os.Stdout, nil)
    logger := slog.New(handler)
    logger.Info("user login", "user", "alice", "ip", "127.0.0.1")
}
// 输出
{"time":"2025-10-16T15:11:02.456Z","level":"INFO","msg":"user login","user":"alice","ip":"127.0.0.1"}

// 3. 上下文与字段
func main(){
    log := slog.With("service", "payment", "version", "v1.0")
	log.Info("transaction processed", "user_id", 123, "amount", 49.9)
}
// 输出
time=... level=INFO msg="transaction processed" service=payment version=v1.0 user_id=123 amount=49.9

// 4. 日志等级控制
func main(){
    handler := slog.NewTextHandler(os.Stdout, &slog.HandlerOptions{
        Level: slog.LevelWarn,
    })
    logger := slog.New(handler)

    logger.Info("not printed")
    logger.Warn("this is printed")

}

# 10.2.3 自定义 Handler（扩展性）

可实现 Handler 接口输出到文件、数据库、远程日志服务：

type Handler interface {
    Enabled(ctx context.Context, level Level) bool
    Handle(ctx context.Context, r Record) error
    WithAttrs(attrs []Attr) Handler
    WithGroup(name string) Handler
}

# 10.3 建议

场景建议
Go < 1.21使用 log 或第三方库（zap、zerolog）
Go ≥ 1.21推荐使用 slog
需要 JSON 输出✅ 使用 slog.NewJSONHandler
需要上下文字段✅ slog.With()
想兼容旧系统可封装兼容层（slog 替换 log.Logger）

# 11.标准输入和标准输出

package main

import (
	"fmt"
	"os"
)

// 从标准输入读入数据
func Scan() {
	fmt.Println("please input a word")
	var word string
	fmt.Scanf("%s\n", &word) //读入第1个空格前的单词，注意要加\n。不能走单元测试
	fmt.Printf("you input:[%s]\n", word)

	fmt.Println("please input two int")
	var word1 int
	var word2 int
	//整数用%d
	fmt.Scanf("%d %d\n", &word1, &word2) //读入多个单词，空格分隔。如果输入了更多单词会被缓存起来，丢给下一次scan
	fmt.Println("sum is:", word1+word2)

	fmt.Println("please input a line")
	content := make([]byte, 100)
	n, err := os.Stdin.Read(content)
	if err == nil {
		fmt.Print("you input:", string(content[:n]))
	}
}

// 标准输出 和 标准错误输出
func Stdout() {
	fmt.Println(os.Stdin.Fd(), os.Stdout.Fd(), os.Stderr.Fd())
	fmt.Println("print") //输出到Stdout
	os.Stdout.WriteString("abc")
	os.Stderr.WriteString("123\n")
	fmt.Fprintf(os.Stdout, "%s\n", "hello") //输出到Stdout
	fmt.Fprint(os.Stderr, "golang")         //输出到Stderr
	// 操作系统级别（用C语言会看到下面的结果，但Go语言不是）
	// Stdin、Stdout、Stderr的文件句柄分别是0、1、2
	// Stdout是行缓冲，即遇到换行符才会输出到终端，在此之前一直是缓存在内存里；Stderr无缓冲，立即输出。
}

	// Scan()
	Stdout()
}

# 12. 用go执行系统命令

package io

import (
	"bytes"
	"fmt"
	"os/exec"
)

// 执行系统命令
func SysCall() {
	//查看系统命令所在的目录，确保命令已安装
	cmd_path, err := exec.LookPath("go")
	if err != nil {
		fmt.Println("could not found command go")
	}
	fmt.Printf("command go in path %s\n", cmd_path)

	cmd := exec.Command("go", "version") //相当于命令go version，注意Command的每一个参数都不能包含空格
	//cmd.Output()运行命令并获得其输出结果
	if output, err := cmd.Output(); err != nil {
		fmt.Println("got output failed", err)
	} else {
		fmt.Println(string(output))
	}

	cmd = exec.Command("python", "D:\\go_project\\go_basic\\io\\hello.py")
	//cmd.Output()运行命令并获得其输出结果
	if output, err := cmd.Output(); err != nil {
		fmt.Println("python execute failed", err)
	} else {
		fmt.Println(string(output))
	}

	cmd = exec.Command("rm", "../data/biz.log")
	var out bytes.Buffer
	var stderr bytes.Buffer
	cmd.Stdout = &out
	cmd.Stderr = &stderr
	//如果不需要获得命令的输出，直接调用cmd.Run()即可
	err = cmd.Run()
	if err != nil {
		fmt.Println(fmt.Sprint(err) + ": " + stderr.String()) //错误信息
	} else {
		fmt.Println(out.String()) //正常的输出结果
	}
}
